激活规则

在朝廷式模型中的阶段归属

- Align: `taizi` + `zhongshu`
- Align（目标对齐）：`taizi` + `zhongshu`
- Architect: `zhongshu`
- Architect（架构设计）：`zhongshu`
- Atomize: `zhongshu`
- Atomize（原子化拆解）：`zhongshu`
- Approve: `menxia`
- Approve（审批）：`menxia`

- Automate（执行）：由 `shangshu` 根据需要调度 `bingbu`、`libu`、`gongbu`、`xingbu`
- Assess: `shangshu` + `libu`
- Assess（评估）：`shangshu` + `libu`

## Document bootstrap
文档初始化规则

When 6A is selected, these files must be created or updated under the current active project's root `docs/` directory before implementation starts.
当选中 6A 时，必须先在当前打开项目根目录下的 `docs/` 目录中创建或更新这些文件，然后才能开始实施。

- `taizi` must put the required-document list into the first task card
- `taizi` 必须把必需文档清单写入首张任务卡
- use the current date as a directory prefix and append a bundle version, for example `docs/2026_03_23_首页优化_v1/`
- 目录名前缀必须使用当前日期并追加版本号，例如 `docs/2026_03_23_首页优化_v1/`
- the `docs/` directory is always relative to the active project root in the IDE
- `docs/` 目录始终相对于当前 IDE 中打开项目的根目录
- when the same feature is revised again, increment the bundle version to `v2`, `v3`, and so on
- 当同一功能再次产出一套工作流文档时，目录版本号递增为 `v2`、`v3` 等
- `zhongshu` must convert the document list into a staged plan
- `zhongshu` 必须把文档清单转成分阶段计划
- `shangshu` must dispatch `libu` to initialize missing files
- `shangshu` 必须调度 `libu` 初始化缺失文件
- if a document already exists, update it instead of creating a duplicate
- 如果文档已经存在，就更新它，而不是创建重复文件
- code execution is blocked until the document skeleton set exists
- 在文档骨架集合存在之前，代码执行被阻断
- any code-changing stage must document target file paths, line ranges, and before/after code context in the workflow docs
- 任何涉及代码修改的阶段，都必须在工作流文档中记录目标文件路径、行号范围以及修改前后代码上下文

## Control rules
控制规则

- do not skip stages
- 不允许跳过阶段
- do not code before Align and Architect are complete
- 在 Align 和 Architect 完成之前，不要进入编码
- if approval fails, return to Architect or Atomize
- 如果审批失败，回退到 Architect 或 Atomize
- each atomic task should map to one executable change unit where feasible
- 在可行情况下，每个原子任务应对应一个可独立执行的变更单元
